Sections — in order

  1. Purpose — what this service does in one paragraph.
  2. Dependencies — what it calls, what calls it. Every named service is a wiki-link.
  3. Known failure modes — bullet list linking to root-causes/ pages.
  4. Incident history — newest first.
  5. Owners — link to the owning team's page.
  6. Notes — open questions, runbooks, anything else.

Rules

  1. File name is the service slug in kebab-case, e.g. checkout-api.md.
  2. Type-specific frontmatter: upstream, downstream, incidents.
  3. Every service named in upstream / downstream must have a page or a stub.
  4. Known failure modes link out to root-causes/ — don't restate the cause inline.
  5. Unknown owner? Write owner: unknown and flag it under Notes.
